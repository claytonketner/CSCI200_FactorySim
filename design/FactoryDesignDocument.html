Kitting Cell Design Document

Purpose/Overview:

        This program is designed to provide GUI support to a kitting cell that assembles kits full of parts for use by other assembly cells.  There will be one server application to interact with the back end of the kitting cell and track the state of each of the cell’s devices.  There will be several client applications that will serve as a GUI for various humans needing to interact with the kitting cell.  These client applications will communicate with the server application to display the state of various cell devices to their respective user.  They will also take input to send configuration information to the cell. Figure 1 below shows what the factory production manager’s monitoring screen will look like.

Figure 1: Concept art of the factory manager’s view of the kitting cell.

Requirements:

        This piece of software will have one server application which will send and receive information to and from the back end of the kitting cell and will communicate similarly with several GUI clients.  The server will hold the “state” information about each device within the cell.  This information may be updated by the front or back end.  Each of the client application GUIs will allow a kitting cell employee to view the state of the device(s) for which they are responsible.  Some GUIs will also allow the user to set configuration information or simulate non-normative scenarios within the factory.  The GUI clients consist of:

  3. This panel will not have any graphics

  5. Kits will be serializable

Manager Classes:

Server:

  1. Server
  1. class implementing a non-GUI server application to coordinate factory clients over a network
  2. Member Constants (all public and static):
  1. Port - networking port that server listens on
  2. SettingsPath - path of file where factory state is saved
  1. Member Data (all private):
  1. serverSocket - server socket used to set up connections with clients
  2. netComms - ArrayList of client connections
  3. wantsFactoryState - ArrayList of booleans indicating whether each client wants to be updated with the factory state
  4. partTypes - ArrayList of Parts that are available to produce
  5. kitTypes - ArrayList of Kits that are available to produce
  6. produceStatus - ProduceStatusMsg storing current kit production status
  7. factoryState - FactoryStateMsg storing current factory state
  8. factoryUpdate - FactoryUpdateMsg storing changes to broadcast to clients on next timer tick
  1. Methods:
  1. Server - constructor for server class
  2. main - instantiates a new Server
  3. actionPerformed - called during timer tick; updates simulation and broadcasts factoryUpdate to clients
  4. msgReceived - handle message received from clients (takes message and NetComm that received the message); generally calls another method to handle the specific message
  5. addPart - takes an AddPartMsg and client index, adds part to partTypes (if valid), sends StringMsg to client indicating success or failure
  6. changePart - takes a ChangePartMsg and client index, changes specified part (if valid and not in production), sends StringMsg to client indicating success or failure
  7. deletePart - takes a DeletePartMsg and client index, deletes part with specified name (if exists), sends StringMsg to client indicating success or failure
  8. listParts - takes client index and sends partTypes to client in a PartListMsg (called when server receives a PartListMsg, but received message is not a parameter because it doesn’t influence behavior of the method)
  9. addKit - takes an AddKitMsg and client index, adds kit to kitTypes (if valid), sends StringMsg to client indicating success or failure
  10. changeKit - takes a ChangeKitMsg and client index, changes specified kit (if valid and not in production), sends StringMsg to client indicating success or failure
  11. deleteKit - takes a DeleteKitMsg and client index, deletes kit with specified name (if exists), sends StringMsg to client indicating success or failure
  12. listKits - takes client index and sends kitTypes to client in a KitListMsg (called when server receives a KitListMsg, but received message is not a parameter because it doesn’t influence behavior of the method)

  14. sendProduceStatus - takes client index and sends produceStatus to client (called when server receives a ProduceStatusMsg)
  15. sendFactoryState - takes client index, sets wantsFactoryState to true for this client, and sends current factoryState to client (called when server receives a FactoryStateMsg)
  16. changeNormative - takes a NonNormativeMsg and client index, breaks or fixes specified item, sends StringMsg to client indicating success or failure
  17. loadSettings - load factory state from file
  18. saveSettings - save factory state to file

Helper Classes:

  1. NetComm
  1. encapsulates networked communication with a single client or server
  2. Member Data (all private):
  1. node - object that is notified when a new message is received
  2. socket - client socket that is endpoint for network communication
  3. in - ObjectInputStream that reads messages related to this client over the network
  4. out - ObjectOutputStream that writes messages related to this client over the network
  1. Methods:
  1. NetComm - constructor that initializes communication with given node (that implements Networked) and given Socket; throws IOException if fails
  2. write - write specified object to output stream
  3. close - close connection
  1. NetCommReader
  1. inner class of NetComm that runs on a separate thread to receive messages
  2. Methods:
  1. run - keep polling for new messages, and forward received messages to network node (this is automatically called when thread starts)
  2. read - returns object read from input stream, or null if no new message, or a CloseConnectionMsg if either client or server disconnected from network

  1. ConnectPanel
  1. panel that all client (manager) classes display to let user connect to server; this class is based on SignInPanel.java in Andrew's HW3 submission
  2. Mockup (labeled components are described in Member Data section):

  1. Member Data (all private):
  1. txtServerAddress - text box to enter server domain name or IP address
  2. btnConnect - button to connect to server
  3. lblActionMsg - label for displaying message or error
  4. action - reference to action listener to notify when user presses connect to server button
  1. Methods:
  1. ConnectPanel - constructor that sets up the panel GUI and stores reference to action listener to notify to connect to server
  2. reset - reset text field and action message text
  3. setActionMsg - display specified action message, allowing HTML formatting
  4. setActionError - display specified action message in red text, allowing HTML formatting
  5. actionPerformed - when button clicked, validate sign in info and notify parent action listener

Interfaces:

  1. Networked
  1. indicates that implementing class is capable of receiving messages across the network
  1. Server class and all client (but not panel) classes implement this
  1. Methods:
  1. msgReceived - handle message received from the network (takes message and NetComm that received the message)
